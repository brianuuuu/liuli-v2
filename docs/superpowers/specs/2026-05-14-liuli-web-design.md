# 琉璃 Web 前端设计文档

## 1. 背景与目标

琉璃 Web 端是个人投资研究工作台，服务于市场信号发现、赛道判断、标的分析、预警处理、组合管理、知识沉淀和系统运营管理。

本设计以 `docs/liuli_system_spec_v6.md` 为唯一架构依据。旧系统前端只作为交互参考，不继承旧目录结构、模块边界或页面组织方式。

首版目标：

- 建立符合新后端模块边界的 Web 前端工程。
- 完成六大业务模块与控制台的一级导航和页面骨架。
- 打通核心列表、详情、任务、报告、公告财报等 API 调用路径。
- 使用 ECharts 提供市场热度、趋势、评分、关系图等可视化能力。
- 第一版实现浅色、深色、跟随系统三种主题模式，默认浅色；功能调试和视觉验收优先浅色。

非目标：

- 不做营销首页。
- 不做 K 线、分时等行情图，后续需要时再引入 `lightweight-charts`。
- 不做复杂主题编辑器或多品牌皮肤。
- 不重写 Android。

## 2. 技术栈

Web 端继续使用 React 技术栈，不切换前端技术方向。

- React 18
- Vite 5
- Ant Design 6
- axios
- dayjs
- echarts
- echarts-for-react

暂不引入：

- `lightweight-charts`：后续做 K 线、分时、成交量等金融行情图时再加。
- 状态管理库：首版优先使用 React 内置状态、Context 和页面级 hooks；如果后续出现跨页面缓存、请求失效和并发状态复杂度，再评估 TanStack Query。

## 3. 信息架构

### 3.1 一级导航

采用方案 A：左侧导航工作台。

左侧一级导航只放六大业务模块和控制台入口：

- 市场雷达
- 赛道发现
- 标的分析
- 预警中心
- 组合管理
- 知识库
- 控制台

一级导航与后端模块边界对齐：

| 一级导航 | 后端目录 | 定位 |
|---|---|---|
| 市场雷达 | `invest_assistant/modules/market_radar` | 信号层 |
| 赛道发现 | `invest_assistant/modules/track_discovery` | 判断层 |
| 标的分析 | `invest_assistant/modules/stock_analysis` | 研究层 |
| 预警中心 | `invest_assistant/modules/alert_center` | 时机层 |
| 组合管理 | `invest_assistant/modules/portfolio` | 行动层 |
| 知识库 | `invest_assistant/modules/knowledge_base` | 认知沉淀层 |
| 控制台 | `invest_assistant/modules/console` | 操作面板 |

控制台不是第七个业务能力模块。它只作为后台操作入口和系统状态聚合入口，具体业务能力仍归属于六大业务模块或 `basic/*`。

### 3.2 二级导航

一级模块内存在多个子功能时，在内容区顶部使用横向 Tabs 切换。左侧导航不承载二级菜单，避免导航树过深。

建议二级结构：

| 一级模块 | 内容区顶部二级导航 |
|---|---|
| 市场雷达 | 总览 / 热度榜 / 信息源 / 标签 / 候选标签 / 关系图 |
| 赛道发现 | 总览 / 候选赛道 / 赛道列表 / 证据链 |
| 标的分析 | 总览 / 标的池 / 评分 / 分析报告 / 对比 |
| 预警中心 | 事件 / 规则 / 处理记录 |
| 组合管理 | 组合 / 持仓 / 风险暴露 / 复盘 |
| 知识库 | 笔记 / Skills / Agents / 反馈日志 |
| 控制台 | 系统状态 / 任务中心 / 报告库 / 公告财报库 / 股票基础库 / 系统配置 / AI 日志 |

页面骨架：

```text
左侧一级导航
└── 内容区
    ├── 顶部系统状态栏
    ├── 当前模块标题与关键操作
    ├── 横向二级导航 Tabs
    └── 当前子模块内容
```

## 4. 布局与视觉风格

### 4.1 整体布局

- 左侧固定一级导航，适合长期工作台使用。
- 顶部状态栏展示当前用户、系统状态、后台任务状态、最近更新时间和主题切换。
- 内容区采用中高密度布局，避免营销页式大 Hero 和装饰性卡片。
- 常用模式为筛选区、表格、图表、右侧抽屉详情组合。

### 4.2 视觉风格

- 默认浅色主题。
- 风格应接近研究工作台和后台管理系统：克制、清晰、信息密度高。
- 使用 Ant Design 组件体系和 theme token。
- 避免大面积单一色系、渐变装饰、悬浮大卡片和纯展示型页面。
- 图表使用 ECharts，浅色主题下优先保证文字、网格线、图例和悬浮提示可读。

### 4.3 主题模式

第一版实现：

- 浅色
- 深色
- 跟随系统

默认浅色。当前功能调试和视觉验收重点放在浅色；深色模式完成基础可用切换，后续再细调视觉效果。

实现方式：

- Ant Design `ConfigProvider` 管理 `defaultAlgorithm` 和 `darkAlgorithm`。
- 主题偏好保存到 `localStorage`。
- 监听系统 `prefers-color-scheme`，支持跟随系统。
- 图表通过统一封装读取当前主题，切换文字色、网格线、背景和 tooltip 样式。

## 5. 页面形态

### 5.1 首页

`/` 是工作台总览，不做营销首页。

首页展示：

- 市场热度摘要
- 待处理预警
- 最近报告
- 最近任务运行状态
- 关键数据更新时间

### 5.2 详情打开方式

首版规则：

| 内容类型 | 详情形态 |
|---|---|
| 列表浏览类数据 | 右侧抽屉详情 |
| 复杂研究对象 | 独立详情页 |
| 新增、编辑、确认动作 | 弹窗或抽屉表单 |
| 报告 | 抽屉预览 + 独立阅读页 |

具体映射：

- 公告财报详情：右侧抽屉，长报告可打开独立阅读页。
- 预警事件详情：右侧抽屉。
- 任务日志：右侧抽屉。
- 报告详情：抽屉预览加独立阅读页。
- 赛道详情：独立详情页。
- 标的详情：独立详情页。
- 组合详情：独立详情页。
- 知识笔记：抽屉预览，编辑用抽屉或弹窗。

### 5.3 空状态

空数据不使用假数据填充。

空状态应说明当前没有数据，并提供下一步操作入口：

- 暂无公告：引导到控制台触发公告拉取任务。
- 暂无报告：引导到任务中心或对应业务模块生成报告。
- 暂无预警：引导创建预警规则。
- 暂无组合：引导新建组合。

## 6. 模块设计

### 6.1 市场雷达

页面目标：帮助用户发现市场正在关注什么。

主要内容：

- 总览：热度摘要、最近信息源数量、热门标签、近期趋势。
- 热度榜：标的、赛道、热词榜单，支持时间窗口筛选。
- 信息源：新闻、公告、政策、舆情等源数据列表。
- 标签：标签列表、新增、编辑、停用。
- 候选标签：审核通过、拒绝、合并。
- 关系图：标的-赛道、标的-热点词关系图。

图表：

- 热度趋势折线图。
- 热词排名条形图。
- 关系图。

### 6.2 赛道发现

页面目标：判断市场热度是否能沉淀为值得长期跟踪的方向。

主要内容：

- 总览：候选赛道数量、升温赛道、证据数量。
- 候选赛道：从市场雷达升温信号生成的候选。
- 赛道列表：已确认跟踪赛道。
- 证据链：新闻、公告、财报和知识库材料的引用。

复杂赛道使用独立详情页，承载多 Tab 信息。

### 6.3 标的分析

页面目标：找出能承接赛道逻辑的公司。

主要内容：

- 总览：关注标的、近期评分变化、待补证据。
- 标的池：股票列表、所属赛道、状态、关注等级。
- 评分：业务、财务、估值、风险等评分维度。
- 分析报告：标的分析报告列表。
- 对比：多标的横向比较。

复杂标的使用独立详情页。

### 6.4 预警中心

页面目标：跟踪价格、估值、公告、热度、组合等异动。

主要内容：

- 事件：预警事件列表、状态、处理备注。
- 规则：新增、编辑、启停规则。
- 处理记录：已处理事件和复盘。

预警事件详情默认右侧抽屉。

### 6.5 组合管理

页面目标：管理持仓、权重和风险暴露。

主要内容：

- 组合：组合列表和基本信息。
- 持仓：持仓明细、成本、权重。
- 风险暴露：赛道、行业、单标的集中度。
- 复盘：组合复盘记录。

组合详情使用独立详情页。

### 6.6 知识库

页面目标：沉淀经验，提炼 Skills，编排 Agent，记录业务反哺。

主要内容：

- 笔记：心得、复盘、原则、错误案例。
- Skills：可复用分析准则。
- Agents：分析流程编排。
- 反馈日志：业务反哺记录。

### 6.7 控制台

页面目标：系统运营管理，不承载投资分析业务逻辑。

主要内容：

- 系统状态：API、数据库、任务、数据源状态。
- 任务中心：任务列表、启停、立即运行、日志。
- 报告库：报告索引、筛选、预览、下载。
- 公告财报库：公告列表、拉取、解析、引用到业务模块。
- 股票基础库：股票主数据管理。
- 系统配置：业务参数和运行配置。
- AI 日志：AI 调用记录查看。

## 8. 数据流与错误处理

### 8.1 API 调用

- 使用统一 axios client。
- API base URL 通过环境变量配置。
- 登录后 token 保存到本地存储，并由请求拦截器附加。
- 401 统一跳转登录。
- 业务错误统一展示 Ant Design message 或 notification。

### 8.2 数据展示

- 列表页面使用统一分页、筛选和刷新模式。
- 后端暂无数据时展示真实空状态。
- 接口失败时展示错误状态和重试入口。
- 对耗时操作展示 loading 和结果反馈。

### 8.3 图表

- 图表组件封装在 `components/charts/`。
- ECharts option 在页面层组合，通用主题和容器逻辑放在组件层。
- 关系图、趋势图、排名图优先可用，首版不追求复杂交互。

## 9. 验收标准

首版 Web 验收重点：

- 左侧一级导航和内容区二级 Tabs 符合已确认结构。
- `/` 是工作台总览。
- 六大业务模块和控制台页面可进入。
- 控制台的任务中心、报告库、公告财报库、股票基础库、系统配置有可用入口。
- 登录、token、401 处理路径可用。
- 核心列表、空状态、错误状态、刷新行为可用。
- ECharts 图表在浅色主题下正常渲染。
- 浅色、深色、跟随系统三种主题切换存在；浅色是默认和首要验收主题。
- `npm` 构建通过。
- 使用浏览器打开本地页面完成基础视觉和交互检查。

## 10. 风险与约束

- 后端部分接口字段可能继续演进，前端 API 类型需要保持局部、可调整。
- 深色模式第一版只要求功能可切换，视觉细节后续完善。
- 不引入 `lightweight-charts`，因此首版不处理专业 K 线。
- 控制台页面路径不代表能力归属，具体业务逻辑不得写入 `console` 前端 API 之外的业务混合层。
- 旧前端只能参考交互，不复制旧目录结构。
